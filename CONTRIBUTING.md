# Contributing to Prompt Engineering Handbook

Thank you for your interest in contributing to PromptForge.

This project exists to help people learn practical Prompt Engineering and AI Solution Design through clear explanations, useful examples, and hands-on labs.

---

## Contribution Philosophy

PromptForge values quality, clarity, and practical usefulness.

A good contribution should make the handbook easier to understand, easier to use, or more useful in real work.

We prefer:

- clear explanations over complex language;
- practical examples over abstract theory;
- tested prompts over random prompt lists;
- useful improvements over large unfinished additions.

---

## Ways to Contribute

You can contribute by:

- fixing typos;
- improving explanations;
- adding examples;
- suggesting new chapters;
- creating hands-on labs;
- improving document structure;
- translating materials;
- sharing real-world AI use cases.

---

## Writing Style

Please write in a style that is:

- clear;
- practical;
- beginner-friendly;
- technically accurate;
- easy to scan;
- free from unnecessary jargon.

Each document should answer three questions:

1. What is this concept?
2. Why does it matter?
3. How can the reader apply it?

---

## Document Structure

Most learning documents should follow this structure:

```markdown
# Title

## Why it matters

## Concept

## Example

## Best practices

## Common mistakes

## Practice task

## Key takeaway
```

This structure may change when needed, but consistency is important.

---

## Prompt Quality Standard

When adding prompts, please make sure they include as many of these elements as possible:

1. Role
2. Context
3. Task
4. Input variables
5. Output format
6. Constraints
7. Quality criteria

A weak prompt gives the model a vague task.

A strong prompt gives the model a clear job, useful context, and a measurable output format.

---

## Example Contribution

Instead of adding this:

```text
Write a marketing plan.
```

Prefer this:

```text
You are a marketing strategist for small online businesses.

Create a 30-day marketing plan for: [business type].

Target audience: [audience]
Main goal: [goal]
Budget: [budget]

Return the plan as a table with:
1. Day
2. Channel
3. Activity
4. Goal
5. Expected outcome

Keep the plan realistic for a solo founder with limited time.
```

Then explain why the improved version is better.

---

## Pull Request Checklist

Before submitting a pull request, check that your contribution:

- [ ] is clear and easy to understand;
- [ ] solves a real learning problem;
- [ ] uses consistent formatting;
- [ ] avoids unnecessary jargon;
- [ ] includes examples when useful;
- [ ] follows the project structure;
- [ ] improves the handbook in a visible way.

---

## What to Avoid

Please avoid:

- random prompt dumps;
- untested claims;
- overly academic language;
- vague advice;
- duplicate content;
- promotional material;
- low-effort AI-generated text without editing.

---

## Project Principle

> Every concept must be useful. Every explanation must be actionable. Every example must be practical.
